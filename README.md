# Django_sendmail Project 4
 
## Описание:

Проект Django - это приложение на Python.

## Техническое задание на разработку веб-приложения с рассылкой сообщений
Создать веб-приложение на Django, которое позволяет пользователям управлять рассылками сообщений для клиентов. Приложение должно включать функциональность для создания, просмотра, редактирования и удаления рассылок, а также отправки сообщений по требованию.

## ЧАСТЬ 1. Разработка сервиса
## 1. Управление клиентами
Реализовать возможность добавлять, просматривать, редактировать и удалять получателей рассылки (клиентов).

Модель «Получатель рассылки»:
````
Email (строка, уникальное).
Ф. И. О. (строка).
Комментарий (текст).
````

## 2. Управление сообщениями
Реализовать возможность добавлять, просматривать, редактировать и удалять сообщения.

Модель «Сообщение»:
````
Тема письма (строка).
Тело письма (текст).
````
## 3. Управление рассылками
Реализовать возможность добавлять, просматривать, редактировать и удалять рассылки.

Модель «Рассылка»:

Дата и время начала отправки(start_time, тип datetime).
````
Указывает, с какого момента рассылка может быть запущена.
Обязательно для заполнения!
````
Дата и время окончания отправки(end_time, тип datetime).
````
Указывает, до какого момента разрешено выполнять отправку.
Обязательно для заполнения!
````
Статус (status, тип строка). 
Принимает одно из значений:
'Создана' — текущая дата раньше даты начала рассылки (start_time).
'Запущена' — текущая дата между start_time и end_time (включительно).
'Завершена' — текущая дата позже end_time.
Рекомендуем вычислять статус динамически при каждом обращении к рассылке.

Сообщение (message, внешний ключ на модель «Сообщение»). Определяет, какое письмо будет отправлено в рассылке.
Получатели (recipients, связь «многие ко многим» с моделью «Получатель»). Список клиентов, которые получат данную рассылку.
````
Валидация:
start_time не может быть в прошлом.
start_time должен быть раньше end_time.
````
## 4. Отправка сообщений по требованию
Реализовать возможность вручную запускать рассылку через интерфейс пользователя и/или командную строку.

Логика работы отправки:

Инициация. При запуске рассылки происходит проверка:
Если текущее время находится между 
start_time и end_time, отправка разрешена.
Если нет — выдаётся сообщение об ошибке.
Определение получателей. Выбираются все клиенты, связанные с этой рассылкой.
Отправка писем. Для каждого клиента:
выполняется отправка письма с помощью 
send_mail(),
в случае успеха создается запись о попытке со статусом 
'Успешно',
при ошибке создается запись со статусом 
'Не успешно' и текстом ошибки.

## 5. Попытки рассылок
Для каждой попытки отправки сообщения необходимо сохранять:

Дата и время попытки (attempt_time, тип datetime). 
Статус (status: 'Успешно', 'Не успешно').
Ответ почтового сервера (server_response, текст).
Рассылка (mailing, внешний ключ на модель «Рассылка»).

Хранение истории попыток позволяет:
отслеживать эффективность рассылок,
формировать статистику для пользователя,
диагностировать ошибки доставки писем.

## 6. Главная страница
На главной странице приложения отобразить следующие показатели:

Общее количество всех созданных рассылок.
Количество активных рассылок — то есть тех, у которых:
текущая дата находится между 
start_time и end_time,
````
Время: [----Прошлое----] [start_time]=====РАБОТА=====[end_time] [----Будущее----]
````
и статус равен 'Запущена'.

Количество уникальных получателей (общее число клиентов в системе).
## ЧАСТЬ 2. Расширение функциональности
## 7. Регистрация и аутентификация пользователей
Реализовать систему регистрации и аутентификации пользователей.
Пользователи должны иметь возможность зарегистрироваться на сайте, подтвердив свой email.
Должна быть реализована функция входа и выхода из системы.
Необходимо предусмотреть возможность восстановления пароля.
## 8. Статистика и отчеты
Необходимо собирать и отображать информацию о количестве успешных/неуспешных попыток рассылок пользователя и количестве отправленных сообщений.

## 9. Ограничение доступа
Пользователи могут управлять только своими рассылками и клиентами.
Менеджеры могут просматривать все рассылки и клиентов, но не могут редактировать или удалять чужие данные.
Описание ролей и прав доступа:

Пользователь
Создание, просмотр, редактирование и удаление своих клиентов и рассылок.
Просмотр статистики по своим рассылкам.
Менеджер
Просмотр всех клиентов и рассылок.
Просмотр списка пользователей сервиса.
Блокировка пользователей сервиса.
Отключение рассылок.
